# Git 워크플로우 가이드

## 커밋 메시지 형식

```
<type>(<scope>): <한국어 설명>

[선택] 본문 — 변경 이유와 맥락을 한국어로 설명

[선택] footer — 관련 이슈 번호 (Closes #123)
```

## 타입 목록

| 타입 | 사용 상황 |
|------|-----------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변경 없는 코드 개선 |
| `test` | 테스트 추가 또는 수정 |
| `docs` | 문서 작성 또는 수정 |
| `chore` | 빌드/설정 변경 (기능 무관) |
| `perf` | 성능 개선 |

## 커밋 예시

```
feat(user): 사용자 프로필 수정 API 추가

기존 조회 전용 API에서 수정 기능을 분리하여
단일 책임 원칙을 준수하도록 개선

Closes #42
```

## 브랜치 전략

| 브랜치 | 용도 |
|--------|------|
| `main` | 배포 가능한 안정 브랜치 |
| `feat/<기능명>` | 새 기능 개발 |
| `fix/<버그명>` | 버그 수정 |
| `chore/<작업명>` | 설정·의존성 변경 |

## PR 체크리스트

```bash
pnpm validate   # 커밋 전 필수 실행
```

- [ ] `pnpm validate` 전체 통과
- [ ] 비즈니스 로직 변경 시 테스트 추가
- [ ] 아키텍처 의존성 규칙 준수
- [ ] `console.log` 제거
- [ ] `.env` 파일 스테이징 여부 확인

## 주의사항

- `--no-verify` 사용 금지
- `.env`, `.env.local` 커밋 금지
- 민감한 정보 코드 하드코딩 금지
- 새로운 환경 변수는 `.env.example`에만 추가
