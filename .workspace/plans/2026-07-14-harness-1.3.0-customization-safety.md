# Harness 1.3.0 — 커스터마이즈 보호 체계 + eacc 방법론 역수입

- **Date**: 2026-07-14
- **Status**: Done (implementation + verification complete; commit pending user approval)

## Goal

eacc-mcp-server 업그레이드에서 발견한 구조적 문제(프레임워크 소유 파일의
로컬 커스터마이즈가 upgrade 시 조용히 덮어써짐)를 근본 수정하고, 그
프로젝트가 실전에서 발명한 방법론 4건을 프레임워크로 역수입한다.
사용자 결정: 1부 전체 + ①②③④ 채택, ⑤는 대안(파일명 유지, `/start`가
plans의 Status 줄을 grep)으로.

## Approach

### A. 업그레이드 보호 체계 (dpkg conffile 패턴)

**A-1. 베이스라인 해시 기록**

`.harness-meta.json`에 새 필드 추가:

```json
{
  "...기존 필드...": "...",
  "baselines": {
    ".claude/commands/start.md": "sha256hex",
    ".claude/settings.json": "sha256hex"
  }
}
```

- **대상**: frameworkOwned + languageSpecific 전체 (= 덮어쓰기 대상 집합).
  bootstrapIfMissing은 생성 후 불가침이므로 베이스라인 불필요.
  `HARNESS-VERSION` 파일 자체는 버전 마커라 분기 없이 항상 갱신 — 베이스라인
  제외.
- **해시 방법**: 디스크에 **실제로 쓴 내용**(치환 완료 후)을 CRLF→LF 정규화한
  UTF-8 바이트의 SHA-256. 치환 파일(needsSubstitution)은 치환 결과를 해시해야
  프로젝트 파일과 비교 가능. LF 정규화는 P2에서 겪은 autocrlf 노이즈 방지.
- **기록 시점**: `setup.ps1`/`setup.sh`가 파일 쓸 때 + `upgrade`가 파일을
  덮어쓸 때(그 파일 항목만 갱신).
- 해시 도구: PowerShell은 `SHA256.ComputeHash(UTF8(정규화 문자열))`,
  bash는 `sha256sum` → 없으면 `shasum -a 256` 폴백(macOS).

**A-2. upgrade 분기 매트릭스** (`upgrade.ps1`/`upgrade.sh`/`upgrade.py` 공통)

| 조건 | 동작 | 베이스라인 | 요약 표기 |
|---|---|---|---|
| 파일 == 새 템플릿 | 아무것도 안 함 (이미 최신). 남아있는 stale `.new` 있으면 삭제 | 갱신 | (조용히 스킵) |
| 베이스라인 있음 · 파일 == 베이스라인 (무수정) | 새 템플릿으로 덮어쓰기 | 갱신 | `updated` |
| 베이스라인 있음 · 파일 != 베이스라인 (로컬 수정) | 파일 보존, 새 템플릿을 `<파일>.new`로 기록 | **갱신 안 함** | `merge-needed` |
| 베이스라인 없음 (pre-1.3 프로젝트) | 기존 동작대로 덮어쓰기하되 ⚠ 표시 | 신규 기록 | `overwritten (no baseline — review with git diff)` |
| 프로젝트에 파일 없음 | 새로 생성 | 신규 기록 | `added` |

- `merge-needed`의 베이스라인 미갱신 근거: 설치된 기준은 여전히 구버전이므로,
  사용자가 병합하기 전까지 다음 upgrade가 `.new`를 다시 제안하는 게 올바름.
- **병합 완료 자동 감지**: 사용자가 `.new`를 반영해 파일이 새 템플릿과
  같아지면 1행(파일==템플릿)에 걸려 베이스라인 갱신 + stale `.new` 삭제 —
  별도 "병합 완료 신고" 절차 불필요.
- `.new` 파일이 이미 있고 내용이 같으면 재기록하지 않음(mtime 노이즈 방지).
- 요약 출력 마지막에 `merge-needed` 목록을 모아 "각 파일과 옆의 `.new`를
  비교해 수동 병합 후 재실행하면 정리됩니다" 안내.
- `.gitignore` 권고: `.new`는 병합 후 삭제되는 임시물이므로 생성 프로젝트
  `.gitignore`에 `*.new`는 넣지 **않는다** — 커밋 diff에 보여야 리뷰된다.

**A-3. ADR-001 재분류**: frameworkOwned → bootstrapIfMissing (ADR은 생성
순간부터 프로젝트의 의사결정 기록 — eacc가 Python 예시로 다듬은 게 증거).
bootstrap 경로에 needsSubstitution 적용 필요: DATE는 meta의 `createdDate`,
없으면 오늘. (현 bootstrap 코드는 오늘 날짜만 치환 — 확장)

**A-4. arch 테스트 확장 포인트**: 프로젝트 전용 arch 테스트 자리를
bootstrapIfMissing으로 시드:

- python: `tests/arch/test_project_rules.py`
- TS: `src/tests/arch/project-rules.test.ts`
- Java: `src/test/java/arch/ProjectRulesTest.java` (BASE_PACKAGE 치환 필요)

내용: 통과하는 빈 골격 + 헤더 주석("사고/설계 결정에서 나온 프로젝트 고유
불변식 테스트는 여기에 — 프레임워크 골격 파일(test_dependencies 등)은
upgrade가 덮어쓸 수 있다. eacc의 RAG lazy-import 테스트가 원형 사례").
languageSpecific의 기존 골격 파일 상단에도 같은 안내 주석 1줄 추가.

### B. eacc 방법론 역수입

**B-1. SessionStart 훅** (①):

- 신규 `harness-core/scripts/status-context.sh`: `.workspace/STATUS.md`
  존재 시 헤더 한 줄 + 전문 출력, 없으면 `exit 0` (eacc
  `worklog-context.sh`의 STATUS 버전을 일반화). 언어 무관이므로
  **frameworkOwned**에 등록 (languageSpecific 아님).
- 3개 언어 팩 `settings.json`의 hooks에 SessionStart 추가:
  ```json
  "SessionStart": [{ "hooks": [{ "type": "command",
    "command": "bash scripts/status-context.sh" }] }]
  ```
  harness-core 폴백 settings.json에도 동일 추가.
- 주의: settings.json은 languageSpecific(팩별 오버레이) — 4개 사본 전부
  (harness-core + 3팩) 일관되게.
- eacc와의 관계: eacc는 자기 스크립트명이 다를 뿐 동작 동일 — 1.3.0
  재적용 시 `.new`로 제안되고 병합은 선택.

**B-2. Key Invariants 섹션** (②): harness-core `AGENTS.md` 템플릿의
Architecture 섹션 앞에 추가:

```markdown
## Key Invariants (do not break)

Project-specific rules distilled from incidents and design decisions —
the highest-value section of this file. When an incident or a non-obvious
design choice happens, add one bullet here (what + why + where enforced).

- (none yet — add the first one when it happens)
```

이 저장소 자체의 AGENTS.md는 프레임워크 개발용이라 해당 없음 — 템플릿만.

**B-3. validate venv/Windows** (③):

- python 팩 `validate.sh`: 최상단에 인터프리터 탐지 —
  `.venv/bin/python` → `.venv/Scripts/python` → `python3` → `python`
  순서로 첫 존재를 `PY`로 잡고 `$PY -m mypy/ruff/pytest` 호출.
  (이번 세션에서 시스템 Python이 잡혀 가짜 mypy 에러 25개를 직접 겪음)
- python 팩 신규 `scripts/validate.ps1`: 동일 로직의 PowerShell판
  (eacc의 것을 참고하되 프로젝트 특화 옵션은 제외한 최소형).
- 매니페스트 python languageSpecific에 `scripts/validate.ps1` 추가.
- TS/Java는 인터프리터 문제가 없어(pnpm/mvn) 이번 범위 제외 — Notes에
  후속 아이디어로만.
- AGENTS.md 템플릿 Validation 섹션에 Windows 줄 추가
  (`pwsh ./scripts/validate.ps1`) — python 팩의 LANGUAGE_RULES가 아니라
  공통 템플릿이므로 문구는 "Windows: validate.ps1 if present" 수준으로.

**B-4. 즉시 문서 동기화 문구** (④): harness-core AGENTS.md Work Journal
섹션에 추가 — "의미 있는 변경을 완료한 **그 턴에** STATUS.md와 영향받는
문서(README, `.env.example` 등)를 같이 갱신한다. 세션 끝(`/done`)을
기다리지 말고, 시키기를 기다리지 말 것." (eacc 원문 규칙의 일반화)

**B-5. plans Status grep** (⑤ 대안): `start.md` 4단계(Read Work State)에
추가 — STATUS.md의 Active plan 외에, `.workspace/plans/*.md`의
`- **Status**:` 줄을 훑어 `In Progress`인 플랜을 모두 나열(STATUS.md가
가리키지 않는 미완료 플랜의 유실 방지). harness-core + 루트 사본 동기화.
파일명 상태 컨벤션(원안)은 기각 — git mv 관리 비용 대비 이걸로 충분.

## Notes

- eacc 재적용: 1.3.0이 나오면 eacc 프로젝트를 다시 upgrade해 베이스라인을
  받게 하는 후속 작업이 자연스럽다(이때부터 eacc의 커스터마이즈 3종이
  자동 보호됨). 이번 범위 아님.
- .sh 계열은 이 샌드박스에서 E2E 불가(기존 Blockers와 동일) — bash -n +
  코드 리뷰까지만, Mac/Linux 실검증은 사용자 후속.
- ⑤ 원안(파일명 상태)은 기각 — git mv 관리 비용 대비 grep 대안으로 충분.
- **별도 발견(이번 플랜 범위 밖, 미수정)**: validate.ps1 검증 중 python
  팩 자체의 사전 존재 lint 이슈 2건 발견 — (1)
  `language-packs/python/tests/arch/test_dependencies.py:102` E501(105자,
  한도 100), (2) `language-packs/python/tests/domain/test_user.py:9`
  F401(`User` 미사용 임포트). 둘 다 이번 세션에서 건드리지 않은 파일이라
  이번 변경으로 생긴 회귀가 아님 — 별도 `/fix` 대상으로 사용자에게 보고.
